🏗️ Security by Design

WinLab's security model follows defence-in-depth principles. Controls are applied at every layer — network, application, database, and runtime — so that no single misconfiguration creates a critical exposure.

Architecture principles

  • Least privilege — API tokens are scoped to specific operations. Database service accounts have read/write access only to required tables. No admin credentials are used at runtime.
  • Fail-closed — authentication middleware rejects requests by default. Features are opted in, not opted out.
  • Input validation at every boundary — all user input is validated at the API layer before any database operation. Prisma's parameterised queries prevent SQL injection by default.
  • Secrets out-of-code — no credentials, API keys, or signing secrets appear in source code or version history. All secrets are injected via environment variables at deploy time.
  • Dependency hygiene — npm audit runs on every deploy. Dependencies are pinned to reviewed versions. No transitive dependency updates without review.

🔐 Data Security

Encryption in transit

All client–server communication uses TLS 1.2+ enforced at the nginx layer. HSTS is set with a 2-year max-age and preload flag. HTTP requests are permanently redirected to HTTPS (301).

Encryption at rest

Database files are stored on encrypted volumes (AES-256 via host provider). Personally identifiable information — name, email, nickname — is additionally encrypted at the application layer before storage, using AES-256-GCM with a key derived from a server-side secret. Even direct database access does not expose raw PII.

Password handling

Passwords are hashed with bcrypt, work factor 12. Plain-text passwords are never stored, logged, or transmitted. OAuth/SSO users (Microsoft, Google) have no password record at all.

JWT / Session management

Authentication tokens are signed with HS256, expire after 24 hours, and are delivered via httpOnly, Secure, SameSite=Strict cookies. Tokens are not stored in localStorage. Refresh tokens are rotated on each use.

Audit logging

All security-relevant events — logins, failed attempts, consent changes, admin actions, payment events — are logged with timestamp, user ID, and anonymised IP (last octet masked). Logs are retained for 90 days. Log access requires admin authentication.

🌐 Infrastructure

Hosting

WinLab runs on dedicated VPS hardware in the EU (Hetzner, Frankfurt). There are no shared application-layer tenants. The server runs Ubuntu 22.04 LTS with unattended security updates enabled.

Network controls

  • Inbound firewall: only ports 80 and 443 are publicly exposed
  • SSH access restricted to key-based authentication only (password auth disabled)
  • Backend API port (3001) is localhost-only, not exposed externally
  • Cloudflare proxy for DDoS mitigation and edge caching

🇪🇺 GDPR Compliance

Legal basis

WinLab processes personal data under GDPR Art. 6(1)(b) (contract performance) for account and billing data, and Art. 6(1)(a) (consent) for analytics and AI training data. Consent is granular, revocable, and recorded with a timestamp and user ID.

Data subject rights

  • Right of access — users can export all their data (JSON) from the account dashboard at any time
  • Right to erasure — account deletion removes all PII within 72 hours. Payment records retained for 7 years (legal requirement)
  • Right to portability — scenario results, certificates, and progress are exportable as JSON or PDF
  • Right to object — ML training consent and analytics consent can be withdrawn independently at any time from Settings
  • Right to rectification — users can update name, email, and profile data directly in the dashboard

Data Processing Agreement (DPA)

A GDPR-compliant DPA is available for all Business and Enterprise customers. Contact enterprise@winlab.cloud to request a countersigned DPA before or during onboarding.

Data transfers

User data is stored exclusively in EU infrastructure (Frankfurt). AI Mentor queries are processed via Anthropic's API (US-based). This transfer is covered by Standard Contractual Clauses (SCCs) and is only activated with explicit user consent. If your organisation requires that all data remain within the EU, AI Mentor can be disabled at the tenant level.

🔑 SSO / SAML 2.0

Supported identity providers

  • Microsoft Azure AD / Entra ID — OAuth 2.0 + OIDC (all plans)
  • Okta — SAML 2.0 (Enterprise plan)
  • Google Workspace — OAuth 2.0 (Enterprise plan)
  • Generic SAML 2.0 IdP (Enterprise plan)

Provisioning

Just-in-time (JIT) provisioning is supported. Users are automatically created on first SSO login with the role defined in the IdP group mapping. SCIM provisioning for automated user lifecycle management is on the Enterprise roadmap (H2 2026).

Forced SSO

Enterprise admins can enforce SSO-only login for their tenant domain. When enabled, password-based login is disabled for all users with matching email domains. This prevents shadow account creation.

Setup time

Azure AD SSO setup takes approximately 10 minutes using the self-service enterprise dashboard. Okta and custom SAML configurations are handled during the dedicated Enterprise onboarding call.

🧱 Tenant Isolation

Enterprise workspaces are fully isolated tenants. User data, scenario progress, leaderboard entries, custom configurations, and audit logs are scoped strictly to the tenant at every layer.

How isolation is enforced

  • Database layer — every query includes a tenant filter enforced by the ORM (Prisma). Raw SQL access is not used in application code.
  • API layer — JWT claims carry the tenant domain. Middleware validates tenant scope on every authenticated request before any data operation.
  • Lab runtime — each lab session runs in an isolated sandboxed environment. Sessions cannot access filesystem paths, environment variables, or network resources of other sessions.
  • Leaderboard & analytics — cross-tenant aggregations are never exposed. Enterprise leaderboards are scoped to the tenant by default.

There is no administrative interface or internal tool that can view cross-tenant data without explicit super-admin elevation with audit logging.

❓ Security Review FAQ

Pre-answered responses to the most common enterprise vendor security questionnaire questions. For a full completed VSA/SIG Lite, contact security@winlab.cloud.

Does WinLab store credit card numbers or payment data? +
No. All payment processing is handled directly by Stripe. WinLab stores only Stripe customer IDs and subscription status. No card numbers, CVVs, or bank account details ever touch WinLab systems.
Can our security team conduct a penetration test of WinLab? +
Yes. Enterprise plan customers have the right to conduct a scoped penetration test of their tenant environment. A testing agreement and scope document must be signed before testing begins. Contact enterprise@winlab.cloud to arrange.
Does WinLab use customer data to train AI models? +
No, by default. AI Mentor uses Anthropic's API. WinLab does not send identifiable user data to Anthropic. Query context is anonymised before transmission. Users must explicitly opt in to ML training data use, and enterprise admins can disable this option for their entire tenant.
What happens to our data if we cancel? +
Upon subscription cancellation, customer data is retained for 90 days to allow data export. After 90 days, all PII and scenario data is permanently deleted. A data export in JSON format can be requested at any time during the retention window. Payment records are retained for 7 years as required by law.
Is WinLab ISO 27001 or SOC 2 certified? +
Not yet. ISO 27001 preparation is in progress (target: 2026). SOC 2 Type II is on the 2027 roadmap. We recognise that these certifications are required for some procurement processes and are actively working toward them. In the interim, we provide our internal OWASP audit results, security controls documentation, and a completed security questionnaire under NDA upon request.
Who has access to our tenant's data? +
Access to production data is limited to the system owner (Giovanni Spata) for infrastructure maintenance and incident response. Access events are logged. No third-party support staff have access to customer data. Enterprise customers can request a right-to-audit session to verify access controls.
Do you have a Data Processing Agreement (DPA)? +
Yes. A GDPR-compliant DPA is available for Business and Enterprise customers. Contact enterprise@winlab.cloud to receive and countersign the DPA. We typically turn around DPA requests within 2 business days.
How are vulnerabilities in dependencies managed? +
npm audit runs as part of every deployment pipeline. Critical and high severity vulnerabilities are patched within 72 hours of disclosure. Medium severity within 14 days. Dependency versions are pinned and updated on a reviewed schedule, not automatically.
